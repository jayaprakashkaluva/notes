**Scope**: architecture and implementation deep-dive, grounded in the source tree at
`C:\workspace\opensource\ClickHouse` (master, commit `83a7f246ddd`, version 26.7.1 per
`cmake/autogenerated_versions.txt`). Every class/file named below exists in this tree; paths are
relative to the repo root unless stated otherwise.

---

## 1. Big picture

ClickHouse is a columnar, vectorized OLAP DBMS written in C++ (with a small Rust component under
`rust/`). Its design is driven by a few uncompromising principles that show up everywhere in the
code:

- **Column-at-a-time execution.** Data flows through the engine as chunks of columns, not rows.
  Operations are dispatched once per chunk (typically tens of thousands of rows), amortizing
  virtual-call and dispatch overhead and enabling SIMD.
- **Immutability + copy-on-write.** Columns are immutable; transformations produce new columns.
  Table data on disk is organized into immutable "parts" that are only created, merged, and
  dropped — never modified in place.
- **Deliberately leaky abstractions.** Interfaces like `IColumn` provide generic operations, but
  hot paths downcast to concrete column types and touch raw memory directly. Genericity is never
  allowed to cost performance (documented explicitly in `docs/en/development/architecture.md`,
  "Leaky abstractions").
- **Shared-nothing distribution.** Servers in a cluster are independent; coordination happens
  through ZooKeeper/ClickHouse Keeper for replication metadata, and through query fan-out for
  distributed reads. There is no global query plan across the cluster (with newer exchange-based
  distributed steps emerging, see §10).
- **One binary.** `programs/main.cpp` builds a single `clickhouse` binary that multiplexes into
  `server`, `client`, `local`, `keeper`, `benchmark`, `format`, `compressor`, `obfuscator`,
  `disks`, `git-import`, `install`, etc. (see `programs/`). Release binaries are wrapped by
  `programs/self-extracting`.

## 2. The data plane: columns, fields, blocks, types

### 2.1 `IColumn` (`src/Columns/IColumn.h`)

`IColumn` is the in-memory representation of a chunk of a column. Key properties:

- It inherits `COW<IColumn>` (`src/Common/COW.h`): columns are reference-counted immutable
  objects with copy-on-write mutation. Code that wants to modify a column calls `IColumn::mutate`,
  which clones only when the refcount > 1. This is what makes block-level "add a result column,
  never modify inputs" cheap and safe across the pipeline.
- Relational primitives live on the interface: `filter` (WHERE/HAVING), `permute` (ORDER BY),
  `cut` (LIMIT), `insertFrom`/`insertRangeFrom` (bulk append), `compareAt`, plus support machinery
  such as `EqualRange` (sorted-run ranges used by multi-column sorting) and `ColumnCheckpoint`
  (rollback of a column and its subcolumns to a prior size after a failed parse — important for
  streaming input formats).
- Concrete layouts: numeric columns are a single contiguous `PODArray`
  (`src/Common/PODArray.h` — a `std::vector` replacement with padding that permits reading
  15 bytes past the end, enabling branch-free SIMD tails, and no default-initialization on
  resize); `ColumnString` is two arrays (bytes + offsets); `ColumnArray` wraps a nested column +
  offsets; `ColumnNullable` wraps a nested column + a null-mask byte column; `ColumnConst` stores
  one value pretending to be N rows; `ColumnLowCardinality` stores a dictionary + positions.
  Allocation is tracked via `AllocatorWithMemoryTracking` so column memory is attributed to the
  query's `MemoryTracker`.
- The header exposes hooks for JIT compilation (forward declarations of `llvm::IRBuilderBase`
  etc.) — comparison/expression compilation is integrated at the column level.

### 2.2 `Field` (`src/Core/Field.h`)

A `Field` is a discriminated union representing a single value (`UInt64`, `Int64`, `Float64`,
`String`, `Array`, ...). It is the slow path: used for constants, metadata, and boundary APIs
(`IColumn::operator[]`), never in per-row hot loops. `Field` deliberately erases type width —
`UInt8..UInt64` all become `UInt64` — so `IDataType` is needed to recover exact semantics.

### 2.3 `Block` and `Chunk` (`src/Core/Block.h`, `src/Processors/Chunk.h`)

A `Block` is a set of `(IColumn, IDataType, name)` triples — a horizontal slice of a table plus
full metadata. Inside the processor pipeline, the metadata is factored out: each port carries a
fixed *header* (a `Block` with types/names but no rows) and the data moving through it is a
`Chunk` — just columns + row count. This split exists because per-chunk copying of names and type
pointers was measurable overhead at small chunk sizes.

### 2.4 `IDataType` and `ISerialization`

`IDataType` (`src/DataTypes/`) is logical-type metadata; it is intentionally decoupled from
`IColumn` (e.g. `DataTypeDateTime` and `DataTypeUInt32` share `ColumnUInt32`). Serialization is
split into a separate hierarchy, `ISerialization`
(`src/DataTypes/Serializations/ISerialization.h`), which handles:

- **Binary bulk serialization with substreams.** Complex types decompose into multiple streams —
  e.g. `Nullable(T)` is a null-map stream plus nested streams; `Array(T)` is offsets + nested.
  This substream model maps 1:1 onto the multiple `.bin` files of a Wide MergeTree part and onto
  the Native wire format.
- **Sparse serialization.** A column may be serialized in `Default` or `Sparse` variant depending
  on data statistics — chosen per part/column.
- **Text serializations** used by formats: escaped, quoted, CSV, JSON — one implementation shared
  by all formats that need that flavor.

Dynamic types (`Dynamic`, `Variant`, `JSON` — see `src/Columns/ColumnDynamic.h`,
`ColumnVariant.h`, `ColumnObject.h` and their serializations) extend the same substream model with
discriminators and dynamically-inferred typed subcolumns.

## 3. IO and compression

### 3.1 `ReadBuffer` / `WriteBuffer` (`src/IO/`)

All byte IO goes through `ReadBuffer`/`WriteBuffer`: a raw buffer + cursor with one virtual method
(`nextImpl`) to refill/flush. Everything else — `readIntText`, `writeEscapedString`, varint codecs
in `ReadHelpers.h`/`WriteHelpers.h` — is inline non-virtual code over the buffer, so parsing and
formatting are effectively virtual-call-free. Implementations compose like decorators: file,
socket, S3, encrypted, limited, concatenated, hashing, compressed. This composition is a core
idiom: e.g. the native protocol wraps a socket buffer in `CompressedWriteBuffer`; encrypted disks
wrap file buffers in encrypting buffers.

### 3.2 Compressed block format and codecs (`src/Compression/`)

Column data (on disk and on the native wire) is framed into *compressed blocks*: a 16-byte
CityHash128 checksum, a codec-tagged header, then the compressed payload; typical uncompressed
frame size is controlled by `max_compress_block_size`/`min_compress_block_size` (~64 KB–1 MB).
Codecs are pluggable via `ICompressionCodec` and composable in chains (`CompressionCodecMultiple`,
i.e. `CODEC(Delta, ZSTD)`), including:

- General purpose: `LZ4` (default), `ZSTD`, `None`, `ZXC`.
- Specialized/time-series: `Delta`, `DoubleDelta`, `Gorilla`, `FPC`, `T64`, `GCD`, `ALP`,
  `Quantized`, lossy `SZ3`.
- `CompressionCodecEncrypted` for at-rest encryption at the codec layer.

The checksum-per-block design (verified on read) is a deliberate defense against silent hardware
corruption; `programs/checksum-for-compressed-block` exists to debug exactly this.

## 4. Query lifecycle: from bytes on a socket to a pipeline

The orchestration lives in `src/Interpreters/executeQuery.cpp` (`executeQueryImpl`). Stages:

1. **Parse.** `parseQuery` (hand-written recursive descent, `src/Parsers/`) with hard limits
   `max_parser_depth` and `max_parser_backtracks`. Output is an `IAST` tree
   (`src/Parsers/IAST.h`). No parser generator is used. Parsers and AST are dumb by design — no
   name resolution, no types.
2. **Log + normalize.** The query is logged (`logQuery`, with secrets masked), settings applied
   from the `SETTINGS` clause, quotas/access checked against `Context`.
3. **Interpret.** `InterpreterFactory` selects an interpreter per statement type. DDL and simple
   statements have small interpreters (`InterpreterDropQuery`, ...). For `SELECT` there are two
   paths:
   - **Analyzer path (default)**: `InterpreterSelectQueryAnalyzer`
     (`src/Interpreters/InterpreterSelectQueryAnalyzer.h`) — AST → `QueryTree` → passes →
     `Planner` → `QueryPlan`.
   - **Legacy path**: `InterpreterSelectQuery` with `ExpressionAnalyzer`, kept behind
     `enable_analyzer = false`.
4. **Build pipeline.** `QueryPlan` (`src/Processors/QueryPlan/QueryPlan.h`) is optimized
   plan-level, then lowered into a `QueryPipeline` (`src/QueryPipeline/QueryPipeline.h`) of
   connected `IProcessor`s.
5. **Execute.** A `PipelineExecutor` (or its pushing/pulling/completed wrappers under
   `src/Processors/Executors/`) runs the graph. The result of interpretation is a `BlockIO` —
   a pulling pipeline for SELECT, a pushing one for INSERT, a completed one for INSERT SELECT.
6. **Finalize + observe.** `logQueryStart` / `logQueryFinish` / `logQueryException` populate
   `system.query_log`, `query_metric_log`, OpenTelemetry spans; the query result cache
   (`src/Interpreters/Cache/QueryResultCache.h`) is consulted/populated here as well.

### 4.1 The Analyzer (`src/Analyzer/`)

The Analyzer replaced ad-hoc AST-based analysis with a typed intermediate representation, the
**QueryTree**. Node kinds are explicit classes: `QueryNode`, `UnionNode`, `JoinNode`,
`ArrayJoinNode`, `TableNode`, `ColumnNode`, `FunctionNode`, `ConstantNode`, `IdentifierNode`,
`MatcherNode` (for `*`/`COLUMNS(...)` expansion), `LambdaNode`, `InterpolateNode`, etc. — each in
its own file under `src/Analyzer/`.

`QueryTreeBuilder` converts AST → QueryTree; `QueryTreePassManager`
(`src/Analyzer/QueryTreePassManager.h`) then runs an ordered list of passes
(`src/Analyzer/Passes/`). The first and by far largest pass performs *resolution*: binding
identifiers to columns/aliases/CTEs, resolving functions and types, expanding matchers. Subsequent
passes are rule-based rewrites (constant folding, `count(*)` optimizations, moving functions out
of `any`, converting `OR` chains to `IN`, etc.). After the pass pipeline, every node is fully
typed — a property the old `ExpressionAnalyzer` never guaranteed.

### 4.2 The Planner (`src/Planner/`)

`Planner` consumes the resolved QueryTree and emits a `QueryPlan`:

- `PlannerJoinTree` builds the FROM/JOIN subtree — deciding storage read steps, join order
  representation, and how each table expression becomes a source step.
- `PlannerActionsVisitor` and `ActionsChain` lower expressions into **`ActionsDAG`** — the DAG of
  computations (inputs, functions, constants, aliases) that backs `ExpressionStep`/`FilterStep`.
  `ActionsDAG` is the currency of expression manipulation: predicate pushdown, filter splitting,
  and projection pruning are DAG splits/merges.
- `PlannerJoins` / `PlannerJoinsLogical` choose join algorithms; `PlannerCorrelatedSubqueries`
  handles correlated-subquery decorrelation; `PlannerAggregation`/`PlannerExpressionAnalysis`
  compute aggregation stages.

### 4.3 `QueryPlan` and its optimizer (`src/Processors/QueryPlan/`)

`QueryPlan` is a tree of `IQueryPlanStep`s: `ReadFromMergeTree`, `ExpressionStep`, `FilterStep`,
`AggregatingStep`, `DistinctStep`, `SortingStep`, `CreatingSetsStep`, `CubeStep`, `FillingStep`,
`LimitStep`, etc. Plan-level optimizations (under `QueryPlan/Optimizations/`) operate before
pipeline construction: pushing filters down, removing redundant sorting/distinct, using storage
sort order (`optimize_read_in_order`), projection selection, and lifting reads into parallel
replicas. Newer, notable steps in this tree:

- `BuildRuntimeFilterStep` — runtime (join-produced) filters pushed into scans.
- `BroadcastExchangeStep` / `GatherExchangeStep` with matching `*SendStep`/`*ReceiveStep` and
  `BlocksMarshallingStep` — building blocks of a genuine distributed exchange layer (see §10).
- `CommonSubplanStep` / `CommonSubplanReferenceStep` — shared subplan (CTE materialization) reuse.
- `AnalyzePlanStats` — statistics-based plan analysis.

Each step's `updatePipeline` appends processors to the growing `QueryPipeline`; header
compatibility between steps is enforced at build time.

## 5. The execution engine (`src/Processors/`)

### 5.1 `IProcessor` — the pipeline node model (`src/Processors/IProcessor.h`)

A processor has N input ports and M output ports; chunks move across ports whose structure
(header) is fixed. The contract is a cooperative state machine driven by two methods:

- `prepare` — O(1), non-blocking; moves chunks between ports and returns a `Status`:
  - `NeedData` — must receive input before proceeding;
  - `PortFull` — output is blocked, downstream must consume first;
  - `Ready` — call `work` (synchronous, CPU-only — the header explicitly forbids sleeping or
    waiting on IO inside `work`);
  - `Async` — call `schedule` to obtain a file descriptor to poll (this is how remote reads
    avoid blocking executor threads);
  - `UpdatePipeline` — the processor wants to mutate the graph at runtime (add/remove
    processors) — used e.g. to build the rest of a pipeline after a subquery result is known;
  - `Finished`.
- `work` — the actual computation on the chunk(s) pulled in `prepare`.

The header documents the processor taxonomy: sources, sinks, one-in/one-out transforms,
accumulating transforms (sort, aggregate), limiting transforms, `Resize` (N→M rebalancing, which
is how `union` and thread fan-out/fan-in work), `Concat`, ordered merges, and forks. Formats are
processors too (`src/Processors/Formats/`): an `IInputFormat` is a source, an `IOutputFormat` is
a sink, which is why `SELECT ... FORMAT JSON` is just more pipeline.

### 5.2 `PipelineExecutor` (`src/Processors/Executors/PipelineExecutor.h`)

The executor builds an explicit `ExecutingGraph` over the processor set in its constructor
(validating that all ports are connected) and then runs a task-stealing scheduler:

- `execute(num_threads, concurrency_control)` runs the graph on up to `num_threads` threads;
  `executeStep` supports single-threaded incremental execution (used by pulling executors).
- Scheduling is edge-driven: when a processor's `prepare` changes port state, neighbors are
  re-prepared; `Ready` processors are queued into `ExecutorTasks` for any worker thread; `Async`
  processors are parked on an epoll-like poller until their fd fires.
- Runtime graph mutation (`UpdatePipeline`) is folded into the same loop.
- Cancellation is first-class and two-tier: `cancel` (user/timeout) versus `cancelReading`
  (stop sources only, let in-flight data drain — used for `LIMIT`-style early termination).

### 5.3 CPU scheduling and admission (`ConcurrencyControl`)

Thread count per query is bounded by `max_threads`, but the global `ConcurrencyControl`
(described in `docs/en/development/architecture.md`, "Concurrency control") imposes a server-wide
soft limit (`concurrent_threads_soft_limit_num`) via **CPU slots**: a query allocates
`[1, max_threads]` slots, is guaranteed at least one, and up-scales only as slots free up. The
executor integrates with this through `ISlotControl` (see the include in `PipelineExecutor.h`),
so degraded concurrency shows up as fewer executor threads rather than OS-level thread thrash.
Workload isolation on top of this (IO and CPU) lives in `src/Common/Scheduler/` (workloads,
resource queues; `WorkloadResources` is referenced directly by the executor header).

### 5.4 Aggregation (`src/Interpreters/Aggregator.h`)

`Aggregator` is the shared engine behind `GROUP BY`:

- Key layout is specialized via `AggregatedDataVariants` / `AggregationMethod` /
  `ColumnsHashing`: fixed-width keys use typed hash tables (with an inlined last-element cache —
  `LastElementCacheStats`), string keys use string hash maps, multi-key falls back to serialized
  keys. Selection is data-driven and informed by `HashTablesStatistics` (remembered sizes from
  prior runs to pre-size tables).
- Aggregate states are placed in `Arena`s; `IAggregateFunction`
  (`src/AggregateFunctions/IAggregateFunction.h`) exposes batched `addBatch`-style methods so the
  inner loop is one virtual call per chunk per function, not per row.
- Hash tables convert to **two-level** variants past a size threshold, enabling (a) parallel
  merging of per-thread results bucket-by-bucket and (b) streaming/partitioned processing.
- Memory-bounded execution: states spill via `TemporaryDataOnDisk`; distributed and spilled
  states are (de)serialized with `NativeWriter`. Aggregate states are also storable as a data
  type (`AggregateFunction(...)`), which is what `AggregatingMergeTree` persists.
- JIT: `Interpreters/JIT/compileFunction.h` — hot aggregate loops and expression DAGs can be
  compiled with LLVM (`compile_expressions`, `compile_aggregate_expressions` settings),
  with `CompiledAggregateFunctionsHolder` caching compiled artifacts.

### 5.5 Joins (`src/Interpreters/`)

Join algorithms are pluggable behind `IJoin`: `HashJoin/` (the default in-memory hash join),
`ConcurrentHashJoin` (sharded build for parallelism), `GraceHashJoin` (partitioned, spilling),
`MergeJoin` / `FullSortingMergeJoin` (sort-based), plus direct joins against key-value storages
and dictionaries. The planner (`PlannerJoins`) picks among them via `join_algorithm`; runtime
filters (§4.3) feed back from the build side into probe-side scans.

## 6. Storage layer

### 6.1 `IStorage` (`src/Storages/IStorage.h`)

A table engine implements `read` (returns processors that produce data — possibly already
partially processed, indicated by `QueryProcessingStage`), `write` (returns a sink), plus
`alter`/`rename`/`drop`/`mutate`. `read` receives query info so the engine can prune with its own
indexes. `StorageInMemoryMetadata` snapshots columns/keys/constraints so DDL is atomic with
respect to running queries. The engine zoo is broad: `StorageMergeTree`, `StorageMemory`,
`StorageBuffer`, `StorageDistributed`, `StorageJoin`, `StorageKeeperMap`, `StorageLog`,
`StorageMaterializedView`, `StorageMerge`, `StorageFile`, object-storage tables under
`Storages/ObjectStorage/` (S3, Azure, HDFS, Local) with data-lake formats (Iceberg/Delta/Hudi in
`Storages/ObjectStorage/DataLakes/`), integrations (Kafka, RabbitMQ, ...), and `StorageArrowFlight`.

### 6.2 MergeTree — the core engine (`src/Storages/MergeTree/`)

`MergeTreeData` (`MergeTreeData.h`) owns the part set and parameters; reading, writing, and
merging are delegated to `MergeTreeDataSelectExecutor`, `MergeTreeDataWriter`, and
`MergeTreeDataMergerMutator` respectively.

**Parts.** Every INSERT sorts its batch by the sorting key and writes an immutable **part**
directory named `partition-id_min-block_max-block_level`. Per the `MergeTreeData.h` header, a
part contains:

- `checksums.txt` — every file with size + checksum (integrity is checked on load/fetch);
- `columns.txt` — column list with types;
- `count.txt` — row count;
- `partition.dat` + `minmax_[Column].idx` — partition value and min/max indexes for partition
  pruning;
- `primary.idx` — the sparse primary index: primary-key values of every `index_granularity`-th
  row (default 8192; adaptive granularity bounded by `index_granularity_bytes`);
- per-column `[Column].bin` (compressed substreams) and `[Column].mrk*` mark files.

**Marks.** A mark is `(offset of compressed block in .bin, offset within decompressed block)` per
granule per substream — the glue that maps "granule N" to byte positions. Loaded via `MarkCache`
(`src/Storages/MarkCache.h`); `primary.idx` via `PrimaryIndexCache`
(`src/Storages/MergeTree/PrimaryIndexCache.h`).

**Part formats** (`MergeTreeDataPartType.h`): `Wide` (file-per-substream — the steady state) vs
`Compact` (all columns interleaved in one `data.bin` + one mark file — for small parts, controlled
by `min_bytes_for_wide_part`); orthogonally `Full` vs `Packed` storage
(`MergeTreeDataPartStorageType`), the latter packing the whole part suitable for object storage.

**Read path.** `MergeTreeDataSelectExecutor`:
1. Prune partitions via `partition.dat`/minmax indexes.
2. Select the part snapshot (parts are refcounted; a SELECT holds its snapshot while merges
   proceed underneath).
3. `KeyCondition` (`KeyCondition.h`) turns the WHERE clause into an interval lattice over the
   primary-key tuple and binary-searches `primary.idx` to produce candidate **mark ranges**.
4. Data-skipping indexes filter granule ranges further: `MergeTreeIndexMinMax`,
   `MergeTreeIndexSet`, `MergeTreeIndexBloomFilter`, `MergeTreeIndexBloomFilterText`,
   `MergeTreeIndexText` (inverted text index), `MergeTreeIndexVectorSimilarity` (ANN, with its
   own `VectorSimilarityIndexCache`).
5. Surviving ranges are split into read tasks executed by `MergeTreeReadPool` variants that
   load-balance granule ranges across threads (and across replicas for parallel replicas);
   PREWHERE columns are read first, the filter applied, and only surviving granules of remaining
   columns are read. The `QueryConditionCache` (`src/Interpreters/Cache/QueryConditionCache.h`)
   memoizes per-part predicate results across queries.

**Write path.** `MergeTreeDataWriter` splits an inserted block by partition, sorts each by the
sorting key, and writes parts atomically: written to a temporary directory, fsynced, then renamed
and committed to the in-memory part set under `DataPartsLock`. Parts move through explicit states
(`MergeTreeDataPartState`: `Temporary` → `PreActive` → `Active` → `Outdated` → `Deleting`).
There is no WAL/memtable — the header comment in `MergeTreeData.h` and the architecture doc are
explicit that MergeTree is *not* an LSM tree; small-insert workloads are instead absorbed by
**async inserts** (server-side batching before parts are formed).

**Merges.** Background merges pick parts within a partition via a heuristic
(`SimpleMergeSelector` and friends) balancing write amplification against part count, and stream
them through merging transforms. Engine flavor determines merge semantics (documented in
`MergeTreeData.h`): `Ordinary`, `Collapsing` (sign-based row cancellation), `Replacing`
(keep-latest, optional version column), `Summing`, `Aggregating` (merge aggregate states),
`Graphite` (rollup). This is the mechanism behind "eventual" updates: correctness must not depend
on merge timing, so all merging engines have query-time counterparts (`FINAL`).

**Mutations.** `ALTER UPDATE/DELETE` are versioned rewrites: a mutation entry gets a block
number; each affected part is rewritten (only affected columns) into a new part with the mutation
version in its name — the same infrastructure as merges (`MergeTreeDataMergerMutator`,
`MutateTask`). Lightweight deletes write a `_row_exists` mask column instead of rewriting data.
On-fly mutations (`AlterConversions`) let SELECTs apply pending renames/updates before the
rewrite lands.

**Background execution.** Merges/mutations/fetches/moves run on `MergeTreeBackgroundExecutor`
(`MergeTreeBackgroundExecutor.h`) — a preemptive executor for `IExecutableTask`s that slices work
into steps so short tasks aren't starved behind huge merges (round-robin over task steps rather
than FIFO over whole tasks).

## 7. Replication (`src/Storages/StorageReplicatedMergeTree.h`)

`ReplicatedMergeTree` implements **asynchronous multi-master physical replication** coordinated
through ZooKeeper/Keeper. The header documents the znode layout precisely:

- `/metadata`, `/columns` — table structure (checked for compatibility on attach);
- `/log/log-...` — the common replication log: entries are `GET` (fetch a part), `ATTACH`,
  `MERGE`, `DROP`, mutations, etc.;
- `/replicas/<r>/queue/queue-...` — each replica pulls log entries into its own queue
  (`queueUpdatingTask`/`pullLogsToQueue`) and executes them (`queueTask`), *reordering when
  needed* (`shouldExecuteLogEntry`);
- `/replicas/<r>/parts` — each replica's committed part set with checksums (the reference against
  which local state is validated; divergence triggers re-fetch, and unexpected local data is
  moved to `detached/`, never deleted);
- `/blocks` — recent block hashes for **insert deduplication** (retries of the same insert are
  no-ops);
- `/block_numbers` — monotone per-partition block allocation (via ephemeral locks,
  `EphemeralLockInZooKeeper.h`) that gives merges well-defined, gapless intervals;
- `/leader_election` — leaders (plural since 20.5) assign merges; `/quorum` — quorum inserts;
- `/mutations` — replicated mutation entries.

Key design decisions:

- **Replication is physical**: only parts travel between replicas (`DataPartsExchange.h`, over
  the interserver HTTP endpoint — `Server/InterserverIOHTTPHandler.h`). Queries are not
  replayed.
- **Merges are deterministic and coordinated**: a leader writes one `MERGE` entry; every replica
  executes the same merge locally producing byte-identical parts (verified by checksums), so
  network cost stays flat; a replica that falls too far behind fetches the merged part instead.
- **Insert path** (`ReplicatedMergeTreeSink.h`): write part locally → register block hash in
  `/blocks` (dedup) → allocate block number → commit part + `GET` log entry in one ZK
  multi-transaction. `insert_quorum` upgrades this to synchronous acknowledgment by N replicas.
- Per-table threads: `ReplicatedMergeTreeRestartingThread` (session recovery),
  `ReplicatedMergeTreeCleanupThread` (log/blocks GC), `ReplicatedMergeTreePartCheckThread`
  (integrity), `ReplicatedMergeTreeAttachThread`; retry discipline is centralized in
  `ZooKeeperRetries.h`, and fault injection into ZK ops (`ZooKeeperWithFaultInjection`) is built
  into the storage's test surface.

`DatabaseReplicated` (§9) lifts the same log/queue pattern from tables to DDL.

## 8. ClickHouse Keeper (`src/Coordination/`)

Keeper is a from-scratch, ZooKeeper-wire-compatible coordination service built on **NuRaft**
(`contrib/NuRaft`) instead of ZAB:

- `KeeperServer` wires the Raft server; `KeeperStateMachine` applies committed entries to
  `KeeperStorage` — the znode tree (with `CompactChildrenSet` for memory-lean child lists and
  `ACLMap` for interned ACLs).
- `Changelog`/`KeeperLogStore` implement the replicated log with compressed rotated files;
  `KeeperSnapshotManager` writes periodic snapshots (optionally to object storage/disks via
  `KeeperContext`).
- `KeeperDispatcher`/`KeeperRequestDispatcher` batch client requests into Raft appends; reads can
  be served locally with linearizability options; `KeeperReadThreadPool` parallelizes reads.
- Front-ends: the ZooKeeper TCP protocol (`Server/KeeperTCPHandler.h`), 4-letter admin commands
  (`FourLetterCommand`), an HTTP surface (`KeeperHTTPHandlerFactory`, readiness/dashboard
  handlers), and Prometheus metrics.
- Runs embedded in `clickhouse-server` or standalone (`programs/keeper`), with tooling:
  `keeper-client`, `keeper-converter` (ZooKeeper snapshot migration), `keeper-data-dumper`,
  `keeper-bench`.

The server side talks to Keeper/ZooKeeper through the common client in `src/Common/ZooKeeper/`,
which abstracts over either implementation.

## 9. Databases, DDL, and catalogs (`src/Databases/`)

A database engine (`IDatabase`) controls table metadata storage and DDL semantics:

- `DatabaseOrdinary` — legacy: metadata as `.sql` files, non-atomic ops.
- `DatabaseAtomic` — default: tables have UUIDs, data lives under `store/<uuid-prefix>/<uuid>/`,
  enabling atomic `RENAME`/`EXCHANGE TABLES` and deferred drops (drop is a rename plus background
  GC once refcounts drain).
- `DatabaseReplicated` (+ `DatabaseReplicatedWorker`) — replicates *DDL* across replicas via a
  Keeper log with per-replica queues, same pattern as table replication; `ON CLUSTER` DDL for
  non-replicated databases goes through `Interpreters/DDLWorker` instead.
- `DatabaseMemory`, `DatabaseDictionary`, `DatabaseFilesystem`, `DatabaseHDFS`, `DatabaseBackup`
  (browse a backup as a database), and mirrors: `Databases/MySQL`, `Databases/PostgreSQL`
  (materialized replicas of foreign databases), `Databases/DataLake` (catalog-backed lake
  databases).

The global catalog (`Interpreters/DatabaseCatalog.h`) maps names/UUIDs to storages and owns the
drop/detach lifecycle.

## 10. Distributed execution

**Classic model** (`StorageDistributed`, `docs/en/development/architecture.md` "Distributed query
execution"): a `Distributed` table is a view over per-shard local tables. SELECT rewrites the
query, sends it to one replica per shard (load-balanced, via `Client/` connections and
`Processors/Sources/RemoteSource`), asks remote nodes to process **up to a mergeable stage**
(`QueryProcessingStage`: e.g. `WithMergeableState` — partial aggregates), and merges locally
(two-level hash-table merging makes the final merge parallel). Distributed INSERT either writes
synchronously or spools per-shard batches to disk and ships them asynchronously. There is no
global optimizer across nodes; subqueries in `IN`/`JOIN` are handled by explicit strategies
(`distributed_product_mode`, `GLOBAL IN/JOIN` broadcasting temporary tables).

**Parallel replicas**: the newer read path treats replicas of a *single* shard as a dynamic work
pool — a coordinator hands out granule-range tasks to replicas
(`Storages/MergeTree/ParallelReplicasReadingCoordinator.h`), so one large query scales across
replicas without resharding.

**Exchange-based execution (emerging)**: the `QueryPlan` now contains genuine exchange operators
— `BroadcastExchangeStep`, `GatherExchangeStep`, with send/receive steps and
`BlocksMarshallingStep`, plus a `Server/DistributedQuery` component — the foundation for
multi-stage distributed plans with data reshuffling (addressing the "no global query plan"
limitation the architecture doc itself calls out).

## 11. Disks, object storage, and tiering (`src/Disks/`)

The storage stack is layered:

- **`IDisk`** — filesystem-like interface (files, directories, reads/writes returning buffer
  objects). Implementations: `DiskLocal`, `DiskEncrypted` (transparent AES over another disk),
  `ReadOnlyDiskWrapper`, `DiskBackup`.
- **`DiskObjectStorage`** (`Disks/DiskObjectStorage/`) — adapts an `IObjectStorage` (S3, Azure,
  HDFS, web, local emulation) into an `IDisk` by pairing it with a **metadata storage**: local
  metadata files map logical paths to lists of immutable remote blobs. Writes create new blobs;
  the metadata layer provides atomic rename/hardlink semantics that object stores lack.
  Transactions (`IDiskTransaction`) batch metadata+data operations. This is what makes "MergeTree
  over S3" work unchanged — parts are just directories on a `DiskObjectStorage`. Zero-copy
  replication (replicas sharing one set of blobs, refcounted in Keeper) builds on the same
  metadata indirection.
- **Volumes and policies** — `IVolume`/`VolumeJBOD` group disks; `StoragePolicy` orders volumes
  into tiers; TTL `MOVE` rules and size thresholds drive background part moves
  (`MergeTreePartsMover`) between tiers (hot NVMe → cold S3).
- **`FileCache`** (`src/Interpreters/FileCache/`) — a segment-based local cache of remote blobs
  with bounded size and eviction policies, sitting under the read buffers of object-storage
  disks; effectively the page cache for S3-backed MergeTree. A userspace `PageCache`
  (`src/Common/PageCache.h`) also exists for cached uncompressed data.

Server-side caches worth knowing as a family: `MarkCache`, `UncompressedCache`
(`src/IO/UncompressedCache.h`), `PrimaryIndexCache`, `QueryResultCache`, `QueryConditionCache`,
`VectorSimilarityIndexCache`, `FileCache`, `PageCache` — all bounded, all observable through
`system.*` tables and events.

## 12. Server, protocols, and sessions (`src/Server/`)

`programs/server` (class `DB::Server`, built on Poco's `ServerApplication`) binds protocol
handlers, each a thread-per-connection front-end:

- **Native TCP** (`TCPHandler.h`) — the richest protocol: hello/version negotiation, queries with
  settings + client info, data as compressed Native-format blocks, server→client streams of
  progress, profile events, logs, totals/extremes, and async cancellation. Backward/forward
  compatibility is maintained via `DBMS_TCP_PROTOCOL_VERSION` feature-gating (the version lives
  in `src/Core/ProtocolDefines.h`; note in `cmake/autogenerated_versions.txt` that it is
  independent of release versioning). Used by `clickhouse-client` and all interserver query
  traffic.
- **HTTP** (`HTTPHandler.h`, `HTTPHandlerFactory.h`) — the universal interface: query in
  URL/body, any format out, sessions, compression negotiation, configurable predefined handlers
  (`HTTPHandlerRequestFilter.h`).
- **Wire-compatible emulations**: `MySQLHandler`, `PostgreSQLHandler` — real protocol
  implementations letting MySQL/Postgres clients connect directly.
- **gRPC** (`GRPCServer.h`), **Arrow Flight** (`Server/ArrowFlight/`), **Prometheus**
  (`PrometheusRequestHandler`), interserver part-exchange (`InterserverIOHTTPHandler`), and the
  Keeper handlers (§8).

Session/query state is carried by the **`Context`** hierarchy
(`src/Interpreters/Context.h`; global → session → query, plus a `background` profile context for
merges/mutations — settings merge in a defined order documented in the architecture doc).
`Context` is the dependency-injection hub: catalogs, caches, pools, access control, cluster
topology, settings.

**Access control** (`src/Access/`): multi-backed `AccessControl` (config-defined users + SQL-driven
RBAC persisted on disk or replicated via Keeper), with roles, SQL grants checked at interpret
time, row policies injected as filters into the plan, quotas, and settings profiles/constraints.

## 13. Threading, memory, and observability

- **Thread pools** (architecture doc, "Threads and jobs"): a singleton `GlobalThreadPool`;
  specialized pools (`ThreadPool` instances) draw from it — IO pool, backups IO pool,
  `BackgroundSchedulePool` for periodic tasks, `MergeTreeBackgroundExecutor` for preemptable
  storage tasks. Every job attaches a `ThreadStatus`; query threads join a thread group so
  profile counters and memory aggregate per query.
- **`MemoryTracker`** (`src/Common/MemoryTracker.h`): hierarchical accounting
  (allocation → thread → query → user → server) enforced at the allocator layer
  (`AllocatorWithMemoryTracking`, `PODArray`); exceeding `max_memory_usage` throws
  `MEMORY_LIMIT_EXCEEDED` as an exception in the allocating thread — queries die, the server
  does not. Spilling operators (sort/aggregate/join via `TemporaryDataOnDisk`) key off these
  limits.
- **Observability is a core feature**: hundreds of `ProfileEvents` counters and `CurrentMetrics`
  gauges; system-log tables written by the server about itself (`system.query_log`,
  `part_log`, `metric_log`, `query_metric_log`, `text_log`, `trace_log` filled by the sampling
  query profiler with resolved stacks); `system.processes`, `system.parts`, `system.replicas`,
  etc. — all implemented as storages under `src/Storages/System/`. OpenTelemetry spans propagate
  through query execution (`query_span` in `executeQuery.cpp`).

## 14. Extensibility pattern: registries everywhere

Every user-visible extension point is a factory with self-registration at startup:
`StorageFactory` (engines declare features: supports TTL, replication, etc.),
`DatabaseFactory`, `FormatFactory` (input/output, streaming vs whole-file, schema inference),
`FunctionFactory` + `IFunction` (vectorized `executeImpl` over a `Block`; constant-folding,
lambda/higher-order support, optional JIT compilability), `AggregateFunctionFactory` +
combinator system (`-If`, `-Array`, `-State`, `-Merge` are *composable transformers* over any
aggregate function — the `-State`/`-Merge` pair is what materialized views and
`AggregatingMergeTree` build on), `TableFunctionFactory`, `DataTypeFactory`,
`CompressionCodecFactory`, `DiskFactory`. Adding a function or format touches exactly one
registration site; nothing central must be edited.

## 16. Cross-cutting judgments a senior reader should internalize

1. **The sparse index is a contract, not an optimization.** Everything — mark files, granules,
   PREWHERE, skip indexes, parallel replicas task granularity — is denominated in granules.
   Point-lookup workloads fight the grain of the whole design (the docs say so explicitly);
   batch scans are the native currency.
2. **Immutable parts turn concurrency into snapshotting.** SELECT vs merge vs insert isolation
   is refcounted part sets, not row locks. The cost is write amplification and the
   merge-scheduling heuristics; the benefit is that the read path is essentially lock-free.
3. **Replication trusts determinism.** Byte-identical independent merges + checksums replace
   shipping merged data. Any nondeterminism in merge code is a replication bug — hence the
   strictness around versioned aggregate-state serialization.
4. **Two analyzer stacks still coexist.** New optimizer/planner work happens on the
   QueryTree/Planner path; `InterpreterSelectQuery`/`ExpressionAnalyzer` remain as the
   `enable_analyzer = false` escape hatch. When reading interpreter code, always check which
   path you're on.
5. **The processor state machine is the concurrency boundary.** `prepare` is serialized per
   connected component, `work` is parallel and CPU-only, `Async` externalizes waiting. Any
   blocking inside `work` (IO, locks held long) directly stalls an executor thread — the code
   comments treat this as a hard rule.
6. **Object storage support was retrofitted through metadata indirection, not a rewrite.**
   `IDisk` over `IObjectStorage` + local metadata preserved every MergeTree invariant (atomic
   rename, hardlink-based mutations) on top of blob stores — a case study in extending a system
   without breaking its core abstractions.
